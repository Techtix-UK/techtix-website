System Architecture

OmniDeploy IT is a centralized, hybrid-cloud platform consisting of two main components: a heavily optimized Python-based server backend, and a lightweight Windows Desktop Agent. Together, they form a unified IT Service Management (ITSM), IT Asset Management (ITAM), and Mobile Device Management (MDM) ecosystem.

Platform Synergy

Unlike traditional workflows where RMM (Remote Monitoring) and PSA (Ticketing) are separate tools, OmniDeploy merges them natively. The Desktop Agent feeds live telemetry directly into the Helpdesk grids, allowing agents to instantly view the health of a machine while resolving a user's ticket.

The Desktop Agent

The OmniDeploy Windows Agent runs as a silent, background SYSTEM service on client endpoints. It acts as the bridge between the physical hardware and the OmniDeploy web portal.

API Authentication

To prevent unauthorized data ingestion, every endpoint must authenticate using an agent_api_key. This key is generated by the SuperAdmin in the Web Portal and baked into the MSI installer or passed via command line during deployment.

The agent communicates exclusively via HTTPS POST requests using Bearer token authorization. If the key is missing or mismatched, the backend instantly drops the connection with a 401 Unauthorized response.

Telemetry Polling (The Check-In Loop)

Once deployed, the agent routinely polls the /api/checkin endpoint. On each poll, it updates the central database with live specifications, ensuring your Asset Inventory is always pixel-perfect.

Data Collected Includes:

  • Hardware IDs: Asset Tag, Serial Number, MAC Address.
  • System Specs: OS Version, CPU, RAM total/available, Storage Drives (free space %), Battery Health.
  • Network: Local IP, Public IP, Domain/Location.
  • Security & Access: Current logged-in user, Uptime, BitLocker encryption status, and AnyDesk ID/Passwords for 1-click remote support.
Automated Risk Alerts

If the agent detects a system uptime exceeding 30 days, or a hard drive nearing maximum capacity, it automatically generates a high-priority "System Alert" ticket in the Helpdesk portal directly assigned to the IT team.

MDM & Software Deployments

The Desktop Agent functions as a Mobile Device Management (MDM) client capable of silent remote software execution.

Through the Staff Web Portal, an IT Agent can queue a software installation to an asset. During the desktop agent's next polling cycle, it queries /api/web/deploy_software. If a task is pending, the agent silently downloads the MSI/EXE, executes it using pre-configured silent arguments (e.g., /qn /norestart), and reports the success or failure back to the asset's audit history.

Web: End-User Portals

The End-User environment provides a frictionless self-service and support request mechanism for clients and non-IT staff.

The Support Portal (/portal)

This is the primary entry point for users requiring assistance. OmniDeploy supports both Guest submissions (no login required) and Authenticated submissions.

  • Account Linking: When a user registers, their email address is matched against the IT Directory. If a profile exists, their historical tickets and assigned hardware are instantly attached to their dashboard.
  • Self-Service: Users can browse Published articles from the Knowledge Base to resolve simple issues (like password resets or VPN connectivity) before submitting a ticket.
  • Ticket Submission: Users select a Help Topic (mapped to IT classifications), set an urgency, and describe the issue. Attachments (screenshots, PDFs) are seamlessly uploaded to the server and linked to the ticket ID.

EULA Signatures (/eula)

OmniDeploy modernizes hardware handovers by digitizing the End User License Agreement (EULA). When an IT Agent issues a laptop, they can trigger an automated email containing a unique EULA link, or hand the user an iPad to scan the device's physical QR code.

The EULA portal displays the exact Make, Model, and Serial Number of the device. The user signs the document using a touchscreen or mouse. The backend captures this signature as base64 data, stamps it with the current date/time, and permanently locks the digital receipt into the asset's profile.

Web: Helpdesk Staff

The Staff Portal (/staff) is restricted to IT personnel. It provides comprehensive control over the IT environment.

Ticketing & Routing

The ticketing dashboard features dynamic queues (All Open, My Tickets, Unassigned, Closed) and high-level KPI cards. The ticket modal is heavily engineered for efficiency:

  • Collision Locking: If Agent A opens a ticket, it is locked. If Agent B opens it, they receive a yellow warning banner to prevent duplicate work or conflicting updates.
  • The Thread: Displays emails, user replies, and internal IT notes. System events (like priority changes or reassignments) are injected directly into the timeline.
  • CC & Routing: External vendors or managers can be CC'd on tickets, receiving all automated updates and outbound replies via email.

Asset Management & QR Actions

The Asset Inventory provides a searchable, sortable view of all hardware communicating with the OmniDeploy server. Clicking an asset reveals deeply technical data ingested by the Desktop Agent, as well as manual fields populated by IT (Purchase Date, Warranty Expiry).

Rapid Dispatch & Barcode Scanning

Built for the "Walk-Up" IT desk, the Dispatch tab provides two highly optimized tools:

  1. Caller Lookup: Type a phone number or name. OmniDeploy queries the Directory, pulls up the user, and dynamically loads their assigned devices into a dropdown, allowing the agent to log a highly detailed ticket in less than 5 seconds.
  2. Hardware Scanner: Utilize a webcam or mobile camera to scan a device's barcode/QR code. Instantly check the device into inventory, or assign it to a user and automatically trigger the EULA email workflow.

Web: SuperAdmin Configuration

Users granted the SuperAdmin role have access to the global configuration settings, dictating how OmniDeploy communicates with the outside world.

Inbound & Outbound Email Automation

OmniDeploy can act as a fully autonomous email client, converting inbound emails to tickets and dispatching updates to users.

SMTP (Outbound): Required for the system to send notifications, ticket replies, and EULA receipts. Enter your mail server host, port, user, and password.

Microsoft Graph API (Office 365)

For modern enterprise environments, OmniDeploy supports the Microsoft Graph API for inbound email fetching (bypassing legacy IMAP constraints).

Azure Configuration Requirement

To use Graph API, you must create an App Registration in Microsoft Entra ID (Azure AD). You must generate a Client Secret and grant the application Mail.ReadWrite permissions with Admin Consent. OmniDeploy requires the Tenant ID, Client ID, Client Secret, and Target Mailbox to function.

Macros & Classifications

Admins can customize platform taxonomies to fit organizational workflows.

  • Private Help Topics: Setting a topic's visibility to Private ensures it is only selectable by IT Staff, hiding it entirely from the public-facing portals.
  • Dynamic Canned Responses: Admins can write template replies. Using bracket syntax, the system dynamically injects data:
// OmniDeploy Dynamic Template Variables
{ticket_id} // Injects the current ticket number (e.g., TKT-10492)
{user_name} // Injects the requester's full name
{agent_name} // Injects the replying IT Agent's name

Security & Licensing

OmniDeploy IT utilizes robust security methodologies to ensure data integrity and platform stability.

Database Schema Auto-Patching

OmniDeploy runs on an embedded SQLite backend. As new features (like Staff Roles) are rolled out, the platform includes self-healing schema logic. If the JavaScript frontend detects a 500-level database crash (due to missing columns from an update), it automatically triggers hidden ALTER TABLE SQL injections to silently patch the database schema and reload the UI without human intervention.

Cryptographic License Activation

The platform requires a valid, cryptographically signed license key to operate. The key dictates the platform's Expiration Date, the Registered Client Name, and which operational modules (Tickets, Assets, Directory, Settings) are unlocked in the UI.

The signature validation process operates entirely locally using Web Crypto and Python Hashlib:

def validate_license(license_key):
    payload_b64, signature = license_key.split('.')
    expected = hmac.new(MASTER_SECRET, payload_b64, sha256).hexdigest()

    if hmac.compare_digest(expected, signature):
        return decode(payload_b64)
    else:
        return False

If the key expires, or the signature is tampered with, OmniDeploy enters a strict lockdown mode. All API endpoints and web views are blocked, and agents are redirected to a forced Activation screen. To renew your license or unlock additional modules, contact Techtix Labs.
